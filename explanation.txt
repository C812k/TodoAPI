1- Models → your database entities (tables). (Example: TodoItem)
2- Contracts -> DTOs:  DTO stands for Data Transfer Object. It's a simple class that defines what data the API sends and receives.
3- AppDataContext -> DbContext (EF Core entry point). Where tables live
4- Interfaces -> service contracts.
5- Services -> business logic + databse calls (CRUD)
6- Controllers -> HTTP endpoints only (thin, no business logic)
7- Mapping -> mapping entity ↔ DTO
8- Middleware -> error handling, logging....



 DTOs give you that control. Think of it as:
  - Model = database shape
  - DTO = API shape

   Interface and Service layer:  This is where we define what operations our API can do (Interface) and how it actually does them (Service).

-> Interface (ITodoService): A contract that says "any Todo service must have these methods." It doesn't contain logic
    just method signatures. This is useful because you can swap implementations later (e.g., switch from SQLite to
    PostgreSQL) without changing the rest of your code.

    An interface in C# is like a menu at a restaurant. It tells you what's available (method names, parameters, return
    types), but not how the food is cooked (no implementation). Any class that "implements" this interface must provide all the listed methods.
    Your Controller will depend on ITodoService (the interface), not TodoService (the class).
    This means you can later swap TodoService for TestTodoService or PostgresTodoService without touching the
    Controller. This is called Dependency Injection (DI) — a core .NET concept.

-> Service (TodoService): The actual implementation  it talks to the database via AppDbContext and does the CRUD
    operations.
